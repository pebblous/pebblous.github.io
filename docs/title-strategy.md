# 한국어/영어 제목 전략 가이드

> **목적**: 구글 검색에 강하고(SEO), 사람이 읽으면 끌리는(CTR) 제목을 체계적으로 만드는 가이드
> **대상**: blog-research(초안) → blog-write(최종) → seo-check(검증) 스킬 체인
> **최종 업데이트**: 2026-07-08

---

## 0. 스타일 정본 — 핵심을 전달하는 보도기사·고급 매거진 (JH 확정, 2026-07-08)

### 0.0 제1 기준 — 제목은 "이 글이 무엇에 관한 글인지"를 알려주는 자리다 (JH 확정, 2026-07-19)

문체보다 먼저 오는 단 하나의 기준: **제목만 읽고도 이 글이 무엇에 관한 글인지 짐작되어야 한다.**
그러려면 mainTitle은 **글의 도메인 주어(무엇에 관한 글인지)를 반드시 포함**한다. 수치 대비·훅·여운을
쓰더라도 그것이 주어를 대체해서는 안 된다 — 훅은 주어 위에 얹는 것이지 주어를 버리는 게 아니다.

- ⛔ **주어 실종 = 탈락** (문체가 아무리 깔끔해도). 예: "3일이 만든 128개, 5개월이 만든 883개"
  → 무엇을 128개 만들었는지(블로그 글), 누가(페블러스 블로그)가 없어 제목만으론 주제 파악 불가.
- ⛔ **수치 대비의 유혹 주의**: "128 → 883" 같은 극적 숫자가 있으면 주어를 버리고 숫자 대비만으로
  제목을 만들려는 드리프트가 반복된다. **수치가 강렬해도 주어는 못 버린다.** 숫자는 주어를 꾸미는
  정보이지 주어의 대체물이 아니다.
- ✅ 판별(자가 점검, 게이트): **제목만 소리 내어 읽고 무슨 글인지 5초 안에 말할 수 있는가.**
  말할 수 없으면 훅이 아니라 주어가 빠진 것이니 다시 쓴다.

이 기준은 §3.1(주어 실종 훅 금지)·§3.3 체크리스트와 같은 규율이며, 여기 §0.0에 최우선으로 못박는다
(과거 §3에만 있어 §0만 본 제목 담당이 약하게 받던 문제를 교정).

### 0.1 문체 기준

제목·부제목·SEO제목·설명글의 문체 기준은 하나다: **핵심을 잘 전달하는 한국 보도기사 스타일,
고급 매거진의 절제된 톤.**

| 기준 | 내용 |
|------|------|
| **참조 매체** | 기술 글 → **전자신문** 헤드라인 결 · 사회/정책/비즈니스 글 → **경향신문** 결 |
| **전체 톤** | 고급 매거진 기사 — 사실이 앞서고, 문장은 절제되고, 꾸밈은 정보일 때만 |
| **역할 분담** | mainTitle = 헤드라인(사실을 완결 구문으로) · subtitle = **리드문**(누가·무엇을·왜 한 줄) · pageTitle = 헤드라인의 검색 변형 |

**⛔ 전 슬롯 공통 금지** (mainTitle·subtitle·pageTitle 모두):

| 금지 | 이유 |
|------|------|
| **따옴표 제목** — 제목 안 "..." / '...' 인용 | 어색함(JH). 인용+반전 템플릿의 뿌리이기도 함 |
| 감탄·미끼·과장 (충격·비밀·N가지 비밀) | 보도기사는 사실로 끌지 장치로 끌지 않는다 |
| §3.1 슬롭 템플릿 (대조공식·줄표/콜론 동격·미완결 여운) | mainTitle만이 아니라 **subtitle·pageTitle에도 적용** |
| 억지 차별화 — 이미 좋은 헤드라인을 두고 슬롯마다 새 문장 발명 | 슬롯 간 표류의 원인. pageTitle은 mainTitle의 **키워드 보강 변형**이면 충분 |

**판별 한 줄**: 이 제목을 전자신문/경향신문 지면에 실어도 어색하지 않은가?
데스크가 고쳐 쓸 것 같으면 다시 쓴다.

## 1. 제목의 3가지 슬롯

페블러스 블로그에서 "제목"은 세 곳에 각각 다른 역할로 들어간다:

| 슬롯 | 필드 | 노출 위치 | 글자 수 | 역할 |
|------|------|----------|---------|------|
| **SEO 제목** | `pageTitle` (`<title>`) | 구글 검색 결과, 브라우저 탭 | 50-60자 (` \| 페블러스` 포함) | 헤드라인의 검색 변형 (키워드 보강) |
| **본문 제목** | `mainTitle` (h1) | 페이지 Hero 영역 | 제한 없음 (권장 20-40자) | **헤드라인** — 사실을 완결 구문으로 |
| **부제목** | `subtitle` | Hero h1 아래 | 제한 없음 (권장 30-60자) | **리드문** — 누가·무엇을·왜 한 줄 |

### 핵심 원칙: 세 슬롯은 같은 사실의 세 판형이다 (따로 발명하지 않는다)

```
mainTitle: "팔란티어가 25년 된 온톨로지를 산업 AI의 무기로 바꿨다"
                 ↑ 헤드라인 — 이 글의 사실을 한 문장으로 (지면에 실려도 자연스러운가?)

subtitle: "데이터 통합의 낡은 개념이 Foundry의 운영 계층이 되기까지"
                 ↑ 리드문 — 헤드라인의 맥락을 보충, 보조 키워드 배치

pageTitle: "팔란티어 온톨로지 분석 — 전통 온톨로지와의 5가지 차이 | 페블러스"
                 ↑ 헤드라인의 검색 변형 — 키워드를 앞세우되 새 이야기를 지어내지 않음
```

SEO 제목은 **검색자의 언어로 같은 사실**을 말하고, 본문 제목은 **지면의 언어로 같은 사실**을 말한다.
슬롯이 다르다고 내용이 달라지면 안 된다 — 표류의 시작이다.

### 1.1 네 번째 슬롯 — OG 이미지 제목 (`og-image-title` 오버라이드)

OG 커버(SNS 카드)에 그려지는 제목의 기본값은 **mainTitle 그대로**(오버라이드 비움), 부제는
**og:description(= meta description)**이다. 별도 카피를 짓지 않는다 — §0 정본이 커버 문구에
자동 상속되게 하는 설계이며, 문구가 여러 벌이면 표류한다.

`og-image-title` 오버라이드를 쓰는 경우는 **단 하나**: mainTitle이 커버 2줄을 넘칠 때,
**같은 사실을 더 압축**한다(대략 24자 안, 조사를 줄인 신문 표제 결). 새 문구 발명 금지 —
헤드라인의 압축본이어야 한다. §0 전 슬롯 공통 금지(따옴표·대조·줄표 동격·미끼)가 동일 적용된다.

- ✅ mainTitle "노이즈로 훈련한 신경망이 자신의 과신을 교정했다" → 커버 "노이즈로 훈련한 신경망이 과신을 교정했다"
- ❌ 커버용 후킹 문구를 따로 창작 ("AI의 자신감은 어디서 오는가")

도구: 제목 검토 콘솔(/admin/titles) 상세 패널에서 편집·미리보기, 시각 규칙(2줄 한도·폰트)은
[`og-image` 스킬](../.claude/skills/og-image/SKILL.md)이 담당.

## 2. SEO 제목 공식 (pageTitle)

### 2.1 고CTR 패턴 6가지

실제 GSC 데이터와 업계 베스트 프랙티스 기반:

| # | 패턴 | 공식 | 예시 | 적합한 글 |
|---|------|------|------|----------|
| 1 | **숫자형** | `[키워드] N가지 [가치]` | "VLA 모델 비교 — 5가지 핵심 벤치마크" | 리스트, 비교, 가이드 |
| 2 | **대시 설명형** | `[후킹 문장] — [구체적 내용]` | "1비트 LLM이 공장에 들어온다 — Bonsai 8B 심층 분석" | 기술 분석, 심층 리포트 |
| 3 | **질문형** | `[키워드]? [답변 암시]` | "데이터 품질 없이 AI가 가능한가? — ISO 5259가 말하는 것" | FAQ, 초보자 대상 |
| 4 | **vs 비교형** | `[A] vs [B] — [판단 기준]` | "팔란티어 온톨로지 vs 전통 온톨로지 — 실무 관점 비교" | 비교 분석 |
| 5 | **시의성형** | `[년도/시점] [키워드] [가치]` | "2026 피지컬AI 데이터 전략 완전 가이드" | 트렌드, 연간 전망 |
| 6 | **결과 제시형** | `[방법]으로 [결과]` | "NVFP4 양자화로 31B 모델을 24GB에서 실행하는 법" | How-to, 튜토리얼 |

### 2.2 SEO 제목 체크리스트

- [ ] **핵심 키워드가 앞쪽 1/3에** — 구글은 앞부분 키워드에 가중치
- [ ] **50-60자 이내** (` | 페블러스` 8자 포함) → 실질 42-52자
- [ ] **구체적 약속** — "심층 분석", "비교", "가이드", "N가지" 등 독자가 얻을 것 명시
- [ ] **mainTitle에서 파생됐는가** — 같은 사실의 검색 변형. mainTitle을 버리고 새 문장을 발명하지 않았는가 (§0)
- [ ] **감정 장치 없음** — 사실과 키워드로 끈다. "진실"·"재발명" 같은 트리거 어휘로 끌지 않는다 (§0 보도기사 기준)
- [ ] **브랜드 접미사**: ` | 페블러스` (KO) / ` | Pebblous` (EN)

### 2.3 한국어 SEO 제목 특수 규칙

| 규칙 | 이유 |
|------|------|
| 영문 고유명사는 원어 유지 | "팔란티어 Ontology" 아닌 "팔란티어 온톨로지" — 한글 검색 의도 매칭 |
| 조사 붙이기로 자연스럽게 | "VLA 모델을", "AI가" — 키워드 스터핑 느낌 방지 |
| 엠 대시(—)는 pageTitle에서만, 최대 1개 | 키워드부와 약속부 분리에만. mainTitle·subtitle의 줄표는 §3.1 슬롭 |
| 물결표(~) 사용 금지 | 비전문적 인상 |
| 따옴표 금지 (§0) | 제목 안 "..." 인용은 전 슬롯 공통 금지 |

## 3. 본문 제목 공식 (mainTitle)

본문 제목은 **이미 페이지에 들어온 사람**이 보는 것. 검색 키워드보다 사람을 끄는 힘이 우선이다.
**단, 그 힘은 극적 장치가 아니라 "무엇이 사실/발견인가"의 구체성에서 나온다.** 은유·반전·대조 같은
장치는 사실이 약할 때 덧씌우는 분장이고, 그게 곧 "AI가 쓴 티"(슬롭)다. 사실을 평이하게 말하라.

> ⚠️ **핵심 교훈 (2026-06): 슬롭은 룰브릭이 만든다.** 과거 이 문서가 "은유/짧은선언/대조"를
> 고성과 패턴으로 권장하자, 발굴·리서치·작성 전 단계가 그 지시를 충실히 따라 같은 슬롭이 양산됐다.
> 그 패턴은 ko-prose-humanizer가 본문에서 감점하는 T1(줄표 동격)·T2(명사형 종결)·T7(번역투 반전)
> 그 자체다. **제목도 본문과 같은 휴머나이저 렌즈를 통과해야 한다.**

### 3.1 ⛔ 금지 슬롭 템플릿 (= AI 티의 정체)

| 금지 패턴 | 나쁜 예 |
|----------|--------|
| **따옴표 제목** (§0, JH 확정) | 제목 안 "..." / '...' — 인용 자체가 어색. 인용+반전의 뿌리 |
| **인용+반전** | "'깨끗한 데이터'라고 말했다, 증명은 없었다" |
| **대조 공식** | "A가 아니라 B" · "같은 진단, 다른 미래" · "코드는 열렸다, 데이터는 아니다" |
| **줄표/콜론 동격 재진술** | "A — B" · "A: B" (제목 안에서 같은 말 두 번) |
| **명사형 미완결 여운** | "~이 사라진 이후" · "~생존 전략" · "~의 조건/과제" · "~지는 이유"(주어 없이 여운만) |
| **미끼(clickbait)** | "그들이 다른 한 가지" · "N가지 비밀" · "충격" |
| **과한 은유 단독** | "사냥꾼의 곳간, 농부의 땅" (무슨 글인지 안 보임) |
| **주어 실종 훅** (2026-07, JH) | "며칠은 이기고 몇 년은 지는 이유" — 무엇이 이기고 지는지 **도메인 주어가 없어** 카드·OG 헤드라인만으로 주제 파악 불가. 대조·여운으로 사람은 끌지만 "무엇에 관한 글인지"를 버렸다 |

> ⛔ 이 금지 템플릿은 **mainTitle만이 아니라 subtitle·pageTitle에도 적용**된다 (§0).
> 2026-07 실사례: mainTitle은 깨끗한데 subtitle이 "데이터가 아니라 초기화 방식이 만든 과신 — KAIST가…"
> (대조공식+줄표)로 새어나감. 슬롭은 막힌 슬롯을 피해 옆 슬롯으로 흐른다.

> ✅ **양성 규칙 — 주어를 반드시 담는다 (2026-07, JH 확정).** mainTitle은 **글의 도메인 주어(무엇에
> 관한 글인지)를 반드시 포함**한다. 대조·훅·여운을 쓰더라도 그것이 **주어를 대체해서는 안 된다** — 훅은
> 주어 위에 얹는 것이지 주어를 버리는 게 아니다. 이는 발굴 프롬프트(`service/blog-service-engine/src/core/pipelines.ts`
> discover 단계 `title`)의 **"글이 무엇에 관한지 주어를 이름 붙인다"** 와 동일 규율이다. 발굴은 이 규율을
> 지키는데 작성·제목-리뷰 단계에서 훅만 남기고 주어를 떨구는 드리프트가 실제 문제였다.
>
> - ❌ "며칠은 이기고 몇 년은 지는 이유" (주어 없음)
> - ✅ "일간 예보는 이기고 장기 예보는 밀리는 **AI 날씨 모델**" (대조 훅 + 주어 유지)
>
> **드리프트 실사례 (2026-07, #711):** 발굴 산출 제목 "AI 날씨 모델이 끝내 못 배우는 느린 기후 변동"
> (주어 있음, 규율 준수) → 작성 단계 mainTitle "며칠은 이기고 몇 년은 지는 이유"(주어 실종)로 흐름.
> 발굴이 옳았고 하류가 흐렸다. 자가 점검: **제목만 소리 내어 읽고 무슨 글인지 5초 안에 말할 수 있는가.**

### 3.2 ✅ 권장 — 완결 명사구가 기본, 서술 종결은 발견일 때만

제목의 기본값은 **글이 무엇에 관한지 이름 붙이는 완결 명사구**다. 글의 *주어*(무엇에 관한 글인지)를
가리켜 제목만 봐도 무슨 글인지 보이게 한다. "~이다/~한다/~했다" 서술(평서문) 종결은 **그 동사가
곧 글의 발견이고 이미지가 생생할 때만** 쓴다. 관성적 "~했다"는 영어 SVO 헤드라인의 번역투라 그 자체가
AI 티다 — 실제 한국 신문 표제도 명사 종결이 더 흔하다.

| 패턴 | 예시 |
|------|------|
| **완결 명사구 (기본)** | "AI 자재명세서(AI BOM)" · "온디바이스 VLM의 현장 이식" · "데이터의 유통기한" |
| **발견-동사 (예외)** | "챗GPT 이전 유니콘들이 밸류의 68%를 잃었다" · "AI가 기상청을 이겼다" |
| **구체 행위 서술 (발견일 때)** | "AI가 환자마다 다른 알츠하이머 진행 지도를 그린다" |

- **판별**: 제목이 "~다"로 끝나면 물어라 — *이 동사가 곧 글의 발견인가?* 아니면 명사구로 주어를 가리키게 고친다.
- ⛔ **동사형 종결의 두 전제 (JH 확정, 2026-07-19)**: 서술 종결을 쓰려면 **둘 다** 충족해야 한다.
  ① **매우 간결할 것** — 장황한 동사형("~하면서 ~한 끝에 결국 ~했다")은 그 자체로 탈락(§3.3 20-40자·완결).
     ⛔ 게이트 수치(JH 2026-07-21): **동사 종결 + 33자 초과 = "장황한 서술형"으로 하드 감점**(census -3).
     "~할 때마다 ~하는 ~이 나왔다"처럼 관형절을 늘어놓은 문장형 헤드라인이 그 대상. 명사구로 압축하면
     같은 사실이 짧고 담백해진다(예: "AI가 답할 때마다 출처에 값을 매기는 콘텐츠 정산 원장이 나왔다"(36자·서술)
     → "질의마다 출처에 값을 매기는 콘텐츠 정산 원장"(명사구)). 발견-동사는 33자 이내로 짧을 때만 통과.
  ② **그것만으로 본문 내용이 짐작될 것** — §0.0 제1 기준. 동사 하나로 주제가 안 보이면 명사구로 되돌린다.
  즉 발견-동사는 "짧고(≤33자), 주어가 살아 있고, 그 동사가 곧 발견"인 삼박자일 때만. 하나라도 빠지면 완결 명사구가 안전하다.
- 여기서 권하는 건 **완결** 명사구다. 미완결 명사형("~전략"·"~이후"처럼 잘린 여운)은 §3.1대로 여전히 금지.
- 은유·수치는 **사실을 더 또렷하게 할 때만** 쓴다(장식이 아니라 정보로). "데이터도 유통기한이 있다"처럼
  한 번에 이해되고 본문 주장과 직결되면 OK.

### 3.3 mainTitle 체크리스트

- [ ] **20-40자**, 한 문장으로 완결
- [ ] §3.1 금지 템플릿에 해당하지 않는가? (특히 인용+반전·대조공식·줄표동격)
- [ ] 제목만 읽고 **무슨 글인지** 구체적으로 짐작되는가? (은유만 있고 내용이 안 보이면 탈락)
- [ ] **키워드 의존 금지** — SEO는 pageTitle이 담당
- [ ] **ko-prose-humanizer T1·T2·T7 렌즈 통과** — 본문과 같은 기준
- [ ] 소리 내어 읽었을 때 자연스러운가? (JH 보이스 정본 `project/DAL/code-painting-essay` — 줄표 0개, 평이·구체)

## 4. 부제목 공식 (subtitle)

subtitle은 **보도기사의 리드문**이다 (§0). 헤드라인(mainTitle)이 말한 사실에
누가·무엇을·왜의 맥락을 한 줄로 보충하고, 보조 키워드를 자연스럽게 배치한다.

### 4.1 패턴 — 리드문처럼 쓴다

| 패턴 | 공식 | 예시 |
|------|------|------|
| **경위형** | `[주체]가 [방법]으로 [결과]하기까지` | "NVIDIA가 FP4 양자화로 대형 모델의 진입장벽을 낮추기까지" |
| **범위형** | `[키워드A]부터 [키워드B]까지` | "온디바이스 추론부터 에지 배포까지" |
| **출처형** | `[기관/저널]에 발표된 [연구/사건]의 [맥락]` | "KAIST가 Nature Machine Intelligence에 발표한 노이즈 사전학습 연구" |

### 4.2 subtitle 체크리스트

- [ ] **§3.1 금지 템플릿에 안 걸리는가** — 대조공식("A가 아니라 B")·줄표 동격·따옴표는 subtitle에서도 금지 (§0)
- [ ] **mainTitle의 동격 재진술이 아닌가** — 같은 말을 다시 풀어쓰면 리드문이 아니라 메아리다. 새 정보(맥락·출처·범위)를 더해야 한다
- [ ] mainTitle에 없는 **보조 키워드 1-2개** 포함
- [ ] mainTitle과 합쳐 읽으면 **기사 첫 문단처럼** 읽히는가?
- [ ] 30-60자 권장 (줄표는 최대 1개, 동격 재진술 용도 금지)

## 6. 스킬 체인에서의 역할

```
blog-research (초안)          blog-write (최종)           seo-check (검증)

```

## 7. 제목 → Executive Summary 일관성 규칙

mainTitle이 던진 약속을 Executive Summary가 즉시 이행해야 한다. 제목과 첫 화면이 따로 놀면 독자가 이탈한다.

### 7.1 원칙: 제목의 핵심 주장이 Executive Summary 안에 있어야 한다

| 요소 | 역할 | 예시 |
|------|------|------|
| **mainTitle** | 호기심 유발, 핵심 발견 한 줄 | "가장 뛰어난 AI가 가장 적게 공개한다" |
| **Executive Summary** | mainTitle의 근거를 수치로 제시 | FMTI 58→40, 84% 학습코드 미공개 |
| **본문 해당 섹션** | 상세 분석 | 5.2 학습 데이터 투명성의 붕괴 |

만약 mainTitle이 "투명성 위기"를 말하는데 Executive Summary에 투명성 관련 수치가 없다면 → **제목과 본문이 단절**된 것. 반드시 수정.

### 7.2 체크리스트

- [ ] mainTitle의 핵심 키워드가 Executive Summary 3문단 안에 등장하는가?
- [ ] mainTitle이 약속한 "발견"의 근거 수치가 Executive Summary에 있는가?
- [ ] stat-card(핵심 수치 카드)에 mainTitle과 관련된 지표가 포함되어 있는가?
- [ ] subtitle의 보조 키워드도 Executive Summary에서 맥락이 설명되는가?

### 7.3 외부 보고서/논문 기반 글의 제목 전략

외부 보고서(Stanford HAI, McKinsey 등)를 다룰 때:

| 규칙 | 이유 |
|------|------|
| **mainTitle에 외부 브랜드명 넣지 않는다** | 페블러스의 독자적 관점이 묻힘 |
| **subtitle에 외부 보고서명을 넣는다** | SEO 키워드 확보 + 출처 명시 |
| **mainTitle은 페블러스 관점의 발견이어야 한다** | 단순 요약이 아닌 해석을 약속 |
| **페블러스/DataClinic 언급은 본문에서만** | 제목에 넣으면 독자층이 좁아짐 |

```
✅ 좋은 예:
mainTitle: "가장 뛰어난 AI가 가장 적게 공개한다"
subtitle: "Stanford HAI AI Index 2026 핵심 해설"

❌ 나쁜 예:
mainTitle: "AI의 지도를 펼치다"             ← 모호, 클릭 이유 없음
mainTitle: "Stanford HAI AI Index 2026 분석" ← 정보형, 후킹 없음
mainTitle: "DataClinic 관점에서 본 AI Index" ← 독자층 축소
```

### 7.4 교훈 사례: HAI AI Index 2026

- **변경 전**: mainTitle "AI의 지도를 펼치다" / subtitle "Stanford HAI AI Index 2026이 말하는 것"
- **문제**: 시적이지만 모호. Executive Summary에 제목과 직결되는 수치 없음
- **변경 후**: mainTitle "가장 뛰어난 AI가 가장 적게 공개한다" / subtitle "Stanford HAI AI Index 2026 핵심 해설"
- **보강**: Executive Summary에 FMTI 58→40, 84% 학습코드 미공개 문단 추가 + stat-card 추가

## 8. 안티패턴 (하지 말 것)

> 7번의 제목→Exec Summary 단절도 안티패턴에 해당한다.

| 안티패턴 | 문제 | 개선 |
|----------|------|------|
| `mainTitle == pageTitle` (완전 동일) | 검색 키워드 보강 기회 상실 | pageTitle = mainTitle의 **검색 변형** (키워드 앞배치 + 브랜드 접미사). 단, **억지로 전혀 다른 문장을 발명하는 것도 안티패턴** (§0) — 같은 사실의 두 판형이어야 한다 |
| 키워드 나열: "AI 데이터 품질 관리 전략 가이드" | 읽히지 않음, 스팸 느낌 | 자연스러운 문장으로 |
| 너무 추상적: "새로운 시대" | 검색 매칭 실패, 클릭 동기 없음 | 구체적 대상 명시 |
| 영어 직역: "Factory Starts Thinking by Itself" | 어색한 영어 | 영어 관용 표현으로 재작성 |
| 제목에 `페블러스` 반복 | `pageTitle` 접미사와 중복 | 본문 제목에서 브랜드 제거 |
| clickbait: "충격! AI가..." | 전문 블로그 신뢰도 훼손 | 약속을 지키는 제목 |
