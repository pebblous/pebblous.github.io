칭화대·하버드·베이징대 연구진이 만든 벤치마크 NatureBench의 결과다. 논문을 재현하는 데서 그치지 말고, 그 논문이 세운 최고 성능을 실제로 넘어서 보라는 주문이었다. 웹 검색을 막은 채 최신 모델 열 종을 겨루게 했다.

더 눈여겨볼 대목은 성공의 성격이다. 이긴 경우도 새로운 과학적 방법을 발명한 것이 아니라, 낯선 과제를 익숙한 지도학습 예측 문제로 옮겨 푼 쪽이 대부분이었다. 이런 공학적 번역을 다 더하면 성공의 82.7%다.

실패의 원인은 통념과 달랐다. 과제 자체를 잘못 이해해 무너진 경우는 3.1%뿐이었고, 대부분은 어떤 방법을 고를지 잘못 짚었거나 연산 예산이 모자란 경우였다. 코드를 짜는 일은 이미 병목이 아니라는 뜻이다.

발견을 가르는 것은 모델의 크기가 아니라 문제를 어떤 데이터와 과제로 세우느냐였다.

## LinkedIn (EN)

The best coding agent available today beat the published record in just 16 of 90 Nature-caliber tasks — 17.8%.

That is the headline from NatureBench, a benchmark built by researchers at Tsinghua, Harvard, and Peking University. The test was not to reproduce a paper but to surpass the state-of-the-art it reported. Ten frontier models competed with web search disabled.

The nature of the wins matters more than the count. Where agents did succeed, they rarely invented new science; most reframed an unfamiliar task as a familiar supervised-learning prediction. Add up those engineering routes and they account for 82.7% of successes.

The failures overturn the usual story. Only 3.1% of failed runs stemmed from misreading the task itself. The rest came down to choosing the wrong method or running out of compute. Writing working code is no longer the bottleneck.

What separates discovery is not model size but how the problem is framed as data and task.

## Facebook (EN)

One model submitted the same task 258 times.

It swapped methods, then swapped again, and posted its best score on the 220th attempt. Even then, it never reached the record the original paper had set.

This is one scene from NatureBench, a benchmark from researchers at Tsinghua, Harvard, and Peking University. They drew tasks from 90 Nature-family papers, handed them to ten of the strongest coding agents available, and asked not for reproduction but for something harder: beat the published record.

The scorecard is quiet. Even the leading model surpassed the record in only sixteen of the ninety tasks. What held my attention longer was the reason for the failures. Only 3.1% failed because the agent misread the task. Almost all the rest chose the wrong method, or ran out of the budget to push it far enough.

So the code almost always ran. What stopped them came earlier: the judgment of "what kind of problem is this, really?"

We tend to say discovery is slow because the models are still small. These ninety tasks point somewhere else. Before waiting for a larger model, they leave you asking how the problem was framed as data and task in the first place.
