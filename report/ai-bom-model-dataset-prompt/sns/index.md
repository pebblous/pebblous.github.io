## LinkedIn (KO)

기업 한 곳이 평균 14개의 AI 도구를 쓰는데, IT가 아는 건 네다섯 개뿐이다.

소프트웨어 자재명세서(SBOM)는 어떤 코드 부품이 들어갔는지를 적는다. 그런데 AI 시스템은 코드만으로 돌지 않는다. 모델 가중치, 학습 데이터셋, 시스템 프롬프트, 가드레일, 점점 더 에이전트와 MCP 서버까지 한데 얽힌다. AI 자재명세서(AI BOM)는 이 부품들과, 그것들이 어떻게 연결되는지까지 한 목록으로 추적한다.

목록에 없는 것은 통제할 수도 감사할 수도 없다. 그 사각지대의 이름이 '섀도 AI'다. IBM에 따르면 섀도 AI가 연루된 침해는 표준 침해보다 평균 67만 달러 더 비쌌고, 탐지에 267일이 걸렸다.

다만 부품을 찾아내는 일과 그 부품을 믿을 수 있게 만드는 일은 다른 작업이다. 보안 스캐너는 데이터셋을 발견하지만, 그 데이터가 오염됐는지 라벨이 틀렸는지 라이선스를 위반했는지는 판정하지 못한다.

데이터셋이 목록에 있다는 것과 그 데이터를 신뢰할 수 있다는 것은 다른 문제다. 감사 불가능한 데이터는 규제 불가능하다. AI BOM이 부품 목록이라면, 데이터 품질과 계보 감시는 그 부품에 붙는 품질 인증서다.

## LinkedIn (EN)

The average company runs 14 AI tools. IT knows about four or five.

A software bill of materials (SBOM) lists the code components that went into a build. But an AI system doesn't run on code alone. Model weights, training datasets, system prompts, guardrails, and increasingly agents and MCP servers all run together. An AI bill of materials (AI BOM) tracks those components — and how they connect — in a single list.

What isn't on the list can't be governed or audited. That blind spot has a name: shadow AI. IBM found that breaches involving shadow AI cost $670K more than the average and took 267 days to detect.

But finding a component and being able to trust it are two different jobs. Security scanners surface the dataset; they don't judge whether it is poisoned, mislabeled, or out of license.

Having a dataset on the list is not the same as being able to trust it. Data you can't audit is data you can't regulate. If the AI BOM is the parts list, data quality and provenance are the certificate of conformity that attaches to each part.

## Facebook (EN)

Ask a security lead what AI is running inside their company right now, and the answer often comes back blurry.

The average company runs about 14 AI tools, and IT usually knows about four or five. The rest hums along somewhere out of view.

That unseen place has a name now. Shadow AI.

For years we've kept lists of which code components went into a build. But AI doesn't run on code alone. There's a model, the data it learned from, a system prompt, and now agents and outside tools all moving together. Look only at the code list, and that whole set of parts stays in the dark.

So people have started talking about an "AI bill of materials" — a single list that also names the models, the datasets, the prompts. But filling it in raises a harder question.

Is having a dataset on the list the same thing as being able to trust it?

Finding a part and knowing whether to trust it are two different jobs. Unless the list also records where the data came from and how it was checked, even a well-kept inventory becomes a quiet false comfort. Data you can't audit is data you can't regulate — that line stayed with me.
