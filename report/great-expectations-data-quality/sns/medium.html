Great Expectations Deep Dive — The First Line of Defense for ML Pipeline Data Quality and Its Limits

The Numbers Behind the Problem

Google deployed its TFX-based data validation system across 700+ ML pipelines and observed the results over 30 days. The finding: 6% of training runs contained actual data errors (Schelter et al., MLSys 2019). At Google's scale, one in seventeen training jobs started with contaminated data.

The problem extends to the benchmarks themselves. MIT CSAIL researchers analyzed 10 major ML benchmark datasets including ImageNet and CIFAR-10. The average label error rate was 3.4%. CIFAR-100 had 6%. QuickDraw reached 10.12% (Northcutt et al., NeurIPS 2021). The datasets we treat as ground truth are themselves corrupted.

Data quality is not a risk. It is a certainty.

What Great Expectations Does Well

Great Expectations (GX) is the most mature open-source framework built to address this problem through engineering. It reached v1.0 GA in August 2024, recording 11,398 GitHub stars and approximately 28.35 million monthly PyPI downloads as of April 2026.

GX operates on a straightforward principle: declare what your data should look like, and let the framework validate automatically. The core components:

  • Expectations — assert-style declarations for data properties. 47 core types in v1.x, down from 300+ in v0.x after a deliberate simplification
  • Data Docs — automated HTML documentation from validation results. The validation logic itself becomes the documentation
  • Checkpoints — pipeline gates that halt execution when data fails validation. Insert them into Airflow, Dagster, or any orchestrator

For structured data — CSV, Parquet, SQL tables — GX handles schema validation, statistical range checks, null detection, and distribution monitoring at production scale. In an Avanade case study, a GX Checkpoint caught an upstream schema change that was silently zeroing out critical features before the data reached model training.

The v0.x to v1.x Shift

Most Korean-language GX tutorials were written for v0.x (2021-2022). Using them with v1.x will produce errors. The key changes:

  • Data source connections: YAML-based configuration replaced entirely by the Python Fluent Datasources API. Block-style Data Sources deleted
  • Checkpoint configuration: "Validations list" approach replaced with "Expectation Bindings"
  • Expectations reduction: 300+ types consolidated to 47 core types, with custom extension support

The v1.x API direction has been well-received by the community. But migration friction is real — start from the official v1.x documentation, not legacy tutorials.

What GX Cannot See

GX's structural limitation is precisely defined: it operates on structured, tabular data. The following ML training data quality problems are outside its architecture:

  • Image visual quality — blur, occlusion, exposure defects in training images. GX can check file paths exist, but not what the images contain
  • Class imbalance — detecting whether a 95/5 class split will collapse model performance requires AI-level understanding, not rule-based checks
  • Label contamination — a "cat" labeled as "dog" in a training set passes every GX Expectation. The metadata is valid. The label is wrong
  • Distribution drift in unstructured data — sensor data distributions shifting between collection batches require statistical methods beyond column-level checks

These are not edge cases. They are the primary quality problems in ML training data. A pipeline that passes every GX Checkpoint can still produce a model trained on fundamentally flawed data.

The Tool Landscape in 2025-2026

Three market shifts are reshaping the competitive landscape:

  • WhyLabs acquired by Apple (January 2025) — the whylogs open-source library continues, but WhyLabs SaaS no longer exists as a standalone product
  • Soda Core v4 adopts Data Contracts (January 2026) — moving beyond YAML checks to formalize cross-team data quality responsibilities
  • dbt's built-in unit testing strengthened (2024) — dbt Core 1.8+ has narrowed the gap, making "dbt-only vs. GX" a real community debate

The right framing is not "which tool is best" but "which layer does each tool own." GX owns source ingestion and ML pipeline validation. dbt Tests own SQL transformation verification. Soda handles production monitoring. They are complementary, not competing.

The Dual-Layer Defense

An AI-Ready data pipeline must answer two fundamentally different questions:

  • Layer 1 (GX): "Does this data conform to the expected format?" — schema, nulls, value ranges, statistical bounds
  • Layer 2 (DataClinic): "Is this data suitable for AI training?" — visual quality, class balance, label integrity, distribution health

What GX cannot catch is precisely what DataClinic detects. They are not alternatives — they are different layers of the same defense.
