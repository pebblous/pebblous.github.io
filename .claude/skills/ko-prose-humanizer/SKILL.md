# ko-prose-humanizer: 한국어 AI 문체 평가·교정

> 페블러스 산문 검수의 마지막 단계. voice-edit이 "어떤 보이스로 쓸지"를 다룬다면
> 이 스킬은 "AI가 쓴 티"라는 표면 질감을 측정·교정한다.

## 핵심 원칙 — 무엇을 바꾸고 무엇을 지키는가

생성형 모델이 쓴 한국어 산문은 사실·논리가 멀쩡해도 **표면 질감**에서 티가 난다.
이 스킬은 그 질감을 진단하고, 내용은 건드리지 않은 채 문장 결만 사람 글로 되돌린다.

내용이 아니라 질감을 고친다. 다음은 **절대 변경 금지**:

- 숫자·통계·날짜·고유명사·인용문(따옴표 안 원문)
- 인용 출처, 링크, DOI, 참고문헌, 각주
- 프런트매터·메타데이터, 목차, 표의 데이터, 이미지 캡션의 사실
- 글의 논리 구조와 소제목 체계, 단락 순서
- HTML 구조·클래스·속성

바꾸는 것은 오직 **문장의 리듬·종결·연결·어휘 선택**이다. 의심스러우면 보존 쪽으로
판단한다. 사실을 "더 매끄럽게" 한다며 수치를 반올림하거나 표현을 바꾸지 않는다.

## 1단계 — 진단 (AI tell 11종)

각 항목을 0~10으로 채점한다(높을수록 AI 티가 강함). 실제 본문에서 증거 문구를 1~2개씩 뽑는다.

### 문장 단위 (T1~T9)

| 코드 | tell | 신호 | 교정 방향 |
|---|---|---|---|
| T1 | **줄표(—) 동격 재진술** | "~이다 — ~다", "옮긴다 —" 처럼 대시 뒤에 같은 말을 다시 풂. 한 글에 수십 회면 위험 | 대시 대부분 제거. 재진술이면 한 문장으로 합치거나 둘 중 하나만 |
| T2 | **명사형 종결 일색** | 문장 끝이 거의 다 "~이다/~다/~점이다/~사실이다". 동사 종결 거의 없음 | 종결을 섞는다. 동사로 끝나는 문장, 의문·짧은 문장 의도적 배치 |
| T3 | **상투 비유·"한 X" 반복** | 좌표·합주·악장·결·층위·화면 같은 비유 남발, "한 줄/한 문장/한 자리/한 좌표" 반복 | 비유는 글 전체에 1~2개만. "한 X"는 대부분 평범한 표현으로 |
| T4 | **메타 예고문** | "옮긴다", "풀어 쓰자면", "짚어 본다", "묶어 본다", "한 발 더 들어가면", "다시 읽어 본다" | 예고 없이 바로 내용으로. 다음에 뭘 할지 굳이 알리지 않음 |
| T5 | **영어 병기 강박** | 거의 모든 개념에 (English) 괄호 병기 | 첫 등장 핵심어만 병기, 나머지 삭제. 한국어 정의·이름 우선, 영어 괄호 보충 |
| T6 | **규칙적 볼드 요약** | 모든 섹션이 볼드 한 줄 요약으로 끝나는 동일 패턴 | 요약 자체는 두되 형식을 깨뜨림. 일부는 볼드 없이, 일부는 통합 |
| T7 | **번역투** | 불필요 수동태, 과한 명사화, "~에 있어서", "~을 부른다", "~을 가능하게 한다", "~라는 진단이다" | 능동·구어체 한국어로. 주어-동사 직결 |
| T8 | **수치·고유명사 과적재** | 한 문장에 F1·κ·배수·연도를 몰아넣어 SEO 키워드처럼 보임 | 핵심 수치만 본문에, 나머지는 분산하거나 괄호·각주로 |
| T9 | **병렬·3단 강박** | "A·B·C", "도덕→학술→법" 식 대구를 모든 문단에서 반복 | 일부는 풀어서 서술. 병렬은 정말 효과적인 곳만 |

### 구조 단위 (T10~T11)

문장을 다 고쳐도 AI 티가 남는다면 골격 문제다. 다음 둘은 단락·글 단위로 본다.

| 코드 | tell | 신호 | 교정 방향 |
|---|---|---|---|
| T10 | **정보 구조 — 덤프 vs 흐름** | 도입부·요약이 결론 대신 수치·고유명사를 한 단락에 쏟아붓는다. "그래서 결론이 뭔데"가 한참 뒤에 나옴. C레벨이 30초에 핵심을 못 잡음 | 결론 먼저(BLUF). 수치는 가장 센 2~3개만 앞에, 나머지는 본문으로. 단락마다 "독자가 여기서 무엇을 얻는가"를 점검 |
| T11 | **자사·결론 연결의 작위성** | 객관 분석 끝에 "그러므로 우리 제품"으로 점프. 외부 권위(논문·인물)를 자사 정당화로 끌어옴. 분석의 신뢰를 홍보가 깎음 | 자사를 분석의 **결론**이 아니라 분석에 대한 **반응**으로 위치. 본문은 분석으로 끝내고, 홍보는 절제된 editor's note로 분리. "정당화한다"가 아니라 "이런 문제에 관심을 둔다" 톤 |

### 영어 산문 추가 점검

영어 글이면 추가로 확인:
- delve / moreover / furthermore / crucial / paradigm shift / underscore
- em-dash 남용 (영어는 한국어보다 dash 빈도가 더 낮은 게 자연스러움)
- rule-of-three 강박 ("not just X, it's Y", "what X means for Y, Z, and beyond")
- "stands as a testament", "in the realm of", "a tapestry of" 같은 상투구

### 점수 해석

- 항목 합계 / 110 (T1~T11). 0.3 미만이면 사람 글에 가까움, 0.5 이상이면 교정 강권.
- T10·T11은 점수가 낮아도 치명적이다. 문장이 다 깔끔해도 구조가 덤프거나 자사 연결이
  작위적이면 결국 AI 티가 난다. 반드시 별도로 확인한다.
- 점수보다 **증거 문구**가 중요하다. 사용자가 "왜?"를 바로 확인할 수 있어야 한다.

## 2단계 — 자동 검출 grep 패턴 (페블러스 캘리브레이션)

진단을 빠르게 시작하기 위해 grep으로 1차 신호를 잡는다. JH 보이스 정본(`code-painting-essay`)과
AI 출력의 차이로 보정된 임계치.

### 한국어 (대상 파일에 실행)

```bash
FILE="<대상 파일 경로>"

# T1 em-dash 빈도 (본문에서 1/500자 이하 권장. 1/300자 이하는 위험)
python3 << 'EOF'
import re
t = open("$FILE").read()
t = re.sub(r'<meta[^>]*>', '', t)
t = re.sub(r'<script[^>]*>.*?</script>', '', t, flags=re.DOTALL)
text = re.sub(r'<[^>]+>', '', t)
text = re.sub(r'\s+', ' ', text)
n = text.count('—'); c = len(text)
print(f"본문 글자: {c}, em-dash: {n}, 빈도: 1/{c//max(n,1)}자")
EOF

# T2 명사형 종결 (위험 신호어 검사)
grep -cE "이다\.|점이다\.|사실이다\.|것이다\." "$FILE"

# T3 "한 X" 반복
for kw in "한 줄" "한 문장" "한 자리" "한 좌표" "한 가지" "한 화면"; do
  c=$(grep -c "$kw" "$FILE"); [ "$c" -gt 3 ] && echo "  $kw: $c (반복 위험)"
done

# T4 메타 예고문
for kw in "옮긴다" "풀어 쓰자면" "짚어 본다" "묶어 본다" "한 발 더 들어가면" "다시 읽어 본다" "다음 표가"; do
  c=$(grep -c "$kw" "$FILE"); [ "$c" -gt 0 ] && echo "  $kw: $c"
done

# T7 번역투
for kw in "에 있어서" "을 부른다" "라는 진단이다" "을 가능하게 한다" "되어진" "된 것이다"; do
  c=$(grep -c "$kw" "$FILE"); [ "$c" -gt 0 ] && echo "  $kw: $c"
done

# 기존 검증과 동시 실행 — voice-edit 공통 항목
echo "위키체:"; grep -cE "본 보고서|규명한다|분석하고자|살펴본다|고찰한다" "$FILE"
echo "광고체:"; grep -cE "혁신적|획기적|놀라운|차세대|게임 체인저" "$FILE"
echo "AI 접속부사:"; grep -cE "흥미롭게도|주목할 만한|결론적으로|이러한 맥락에서" "$FILE"
```

### 영어 (EN 글)

```bash
# delve, moreover 등 상투구
for kw in "delve" "moreover" "furthermore" "crucial" "paradigm shift" "underscore" "leverage" "stands as a testament" "in the realm of" "It is worth noting"; do
  c=$(grep -ci "$kw" "$FILE"); [ "$c" -gt 0 ] && echo "  $kw: $c"
done
```

### 정본 캘리브레이션 비교 (반드시 함께 실행)

```bash
# JH 보이스 정본 — 비교 기준선
python3 << 'EOF'
import re
for path in [
  "project/DAL/code-painting-essay/ko/index.html",  # JH 직접 글
  "<대상 파일>",                                       # 검사 대상
]:
    t = open(path).read()
    t = re.sub(r'<meta[^>]*>|<script[^>]*>.*?</script>', '', t, flags=re.DOTALL)
    text = re.sub(r'<[^>]+>', '', t)
    text = re.sub(r'\s+', ' ', text)
    n = text.count('—'); c = len(text)
    print(f"{path}: chars={c}, dashes={n}, 1/{c//max(n,1) if n else 0}자")
EOF
```

## 3단계 — 교정

위 11종을 순서대로 적용하되, 한 문장을 고칠 때 여러 tell이 겹치면 한 번에 처리한다.

교정 후 자가 점검:
- 소리 내어 읽었을 때 문장 길이와 종결이 다양한가?
- 대시·"한 X"·메타 예고문이 눈에 띄게 줄었는가?
- 삭제·변경한 곳에서 사실/수치/인용/링크가 그대로 보존됐는가?
- 글쓴이의 원래 관점·강조점이 유지됐는가? (저자 목소리를 내 목소리로 덮지 않는다)

**과교정 금지.** 모든 대시를 0으로, 모든 병렬을 해체할 필요는 없다. 사람도 쓴다.
목표는 "규칙적 반복"을 깨는 것이지 기법 자체를 박멸하는 게 아니다.

## 4단계 — 출력 포맷

ALWAYS 다음 순서로 출력한다:

```markdown

| ... | ... | ... | ... |

> 전체를 표로 만들지 말 것. 패턴을 보여주는 대표 사례만.

## 교정 원문

- 게시 가능한 완성본 전체. 긴 글이면 파일로 저장하고 경로를 제시.
- 프런트매터·표·참고문헌·링크는 그대로 옮기고 산문만 교정.
```

`_workspace/[글-slug]/voice_humanized.md`

짧은 글이면 인라인으로 출력한다.

## 적용 예시 (페블러스 실제 사례)

### Evans Nature 2026 — 117 → 53 em-dash 감소

**T1+T7 (전):**
> 외로운 군중이라는 표현은 즉시적 직관을 부른다. AI 시대의 인기 주제 위에서 일어나는 일이
> 정확히 이것이라는 진단이다.

**(후):**
> '외로운 군중'은 직관적으로 와닿는 말이다. AI 시대 인기 주제에서 벌어지는 일이 바로 이렇다.

**T2+T3 (전):**
> Evans 2026은 그 11년 thesis의 정점이다. 4,130만 편이라는 실증의 규모가 한 논문에 모였다.

**(후):**
> Evans 2026은 그 11년 연구가 다다른 지점이다. 4,130만 편이라는 규모를 한 논문이 다뤘다.

**T4 (전):**
> 같은 단락의 결론을 옮긴다 — "..."

**(후):**
> 같은 단락은 이렇게 끝난다. "..."

**T11 (전):**
> 4부작 다음의 메타·과학사회학 좌표로서, 이번 글은 페블러스를 "데이터 진단·합성 기업"에서
> 과학 다양성 인프라 카테고리 리더로 격상한다.

**(후, Editor's Note로 이동):**
> *편집자의 노트.* 데이터와 AI의 편향을 측정하는 일이라면 페블러스는 늘 구미가 당긴다.
> 그게 과학에서 벌어지는 문제라면 더욱 그렇다. 이 글은 그간의 AI 거버넌스 4부작에 이어
> '지식의 다양성'에 대한 질문에 답해 보려는 다섯 번째 글이다.

## 다른 스킬과의 관계

| 스킬 | 역할 | 호출 시점 |
|------|------|----------|
| **voice-edit** | 어떤 보이스(default·reflective·sns-cover)로 쓸지 결정 | 글 작성 단계 또는 보이스 변경 시 |

**호출 순서 권장**: 작성 → text-reinforce → voice-edit (필요 시) → **ko-prose-humanizer** → blog-polish → seo-check

## 5단계 — 파이프라인 자동 연결

`blog-produce`, `report-produce`, `report-produce-express`의 자기검증 단계에서 자동 호출.

### blog-produce / report-produce Phase 5 (자기검증)
