# voice-essay: JH 보이스 에세이 리라이트

## 핵심 — "날카롭되 따뜻하다"

⛔ "짧은 문장"만 강조하면 건조하고 차가운 테크 리뷰가 된다.
JH의 본질은 **사색하는 엔지니어의 에세이**다.

## 정본 (반드시 Read)

실행 전 이 두 파일을 반드시 읽어라:

1. `project/DAL/code-painting-essay/ko/index.html` — JH 본인이 쓴 대전 비엔날레 기고문. **이 글의 톤이 기준이다.**
2. `memory/user_writing_voice.md` — JH 보이스 7원칙 상세

## JH 보이스 7원칙

| # | 원칙 | 핵심 | 위반 시 증상 |
|---|------|------|------------|
| 1 | **개인적 서사** | "나"가 있는 글. 직접 경험, 직접 발견 | 무인칭 논문체 |
| 2 | **따뜻한 비유** | 일상 감각으로 번역. 빈정거림 아닌 공감 | 테크 리뷰어 냉소 |
| 3 | **사색적 호흡** | 짧은 선언과 긴 사색이 교차. 리듬 | 전부 짧으면 건조 |
| 4 | **담대한 질문** | 답 전에 질문. 질문 자체가 메시지 | 주장만 나열 |
| 5 | **겸손한 고백** | 한계·편견·놀라움을 솔직히 | 거짓된 객관 |
| 6 | **구체적 숫자** | 이름, 연도, 숫자로 설득 | 추상 서술 |
| 7 | **절제된 연결** | 페블러스는 마지막에, 짧게, 자연스럽게 | 광고처럼 튀는 언급 |

## Before/After 예시

### ❌ 위키피디아체 (AI 기본 출력)
```
NanoClaw(v1.2.12)는 21개 소스 파일, 5,526줄이라는 극단적으로 응축된
코드베이스 위에 컨테이너 격리 기반 멀티채널 AI 에이전트를 구현한
오픈소스 프레임워크다.
```

### ❌ 테크 리뷰어 (v1에서 범한 실수 — 짧지만 차갑다)
```
21개 파일. 5,526줄. 이것이 NanoClaw의 전부다.
플러그인 시스템? 없다. 설정 파일? 없다.
```

### ✅ JH 보이스 (날카롭되 따뜻하다)
```
21개 파일, 5,526줄. 처음 소스를 열었을 때, 이 안에 에이전트 하나의
전체 생애가 담겨 있다는 사실이 믿기지 않았다.

메시지를 받고, 생각하고, 답하는 — 그 순환의 모든 것이 여기에 있다.
파일 하나하나를 따라가 보면, 설계자가 내린 수십 번의 선택이 보인다.
어떤 선택은 우아하고, 어떤 선택은 대가를 치른다.
이 보고서는 그 선택들을 하나씩 열어보는 과정이다.
```

### ✅ JH 보이스로 한계를 말할 때
```
작게 만든 것에는 이유가 있지만, 대가도 있다.

NanoClaw에는 벡터 검색이 없다. 반성적 메모리도 없다.
오래 대화하면 컨텍스트 윈도우가 바닥난다.
에이전트 메모리 서베이(arXiv:2603.07670)가 제안하는
write-manage-read 루프를 떠올려 보면, NanoClaw는
아직 '기억하는 법'을 배우지 못한 셈이다.
```

## 금지 패턴

| 패턴 | 증상 | JH 대안 |
|------|------|---------|
| 논문체 | "본 보고서에서는 ~을 규명한다" | "이 글은 ~을 열어본다" |
| 과장 | "혁신적인", "획기적인" | 사실만. "27,690 스타가 말해준다" |
| 빈정거림 과다 | "X? 없다. Y? 없다." (3회 이상) | 따뜻한 서술로: "X가 없다. 이것은 의도된 선택이다" |
| 전부 짧은 문장 | "A다. B다. C다." 연속 | 짧은 선언 후 긴 사색으로 전환 |
| 수동태 남발 | "~되었다", "~된 것이다" | 능동태: "~했다", "~이다" |
| 감탄 | "!" 남발 | 마침표로 충분하다 |

## 실행 절차

### Step 1: 정본 읽기
```
Read project/DAL/code-painting-essay/ko/index.html (line 750~870)
Read memory/user_writing_voice.md
```
에세이의 톤을 체화한 후 대상 파일을 읽는다.

### Step 2: 대상 읽기 + 리라이트 대상 식별
- `<p class="themeable-text ...">` 문단들
- `<div class="key-insight">` 안의 문단들
- Executive Summary 전체
- 각 섹션의 도입 문단
- h3 소제목 (논문체면 에세이체로)

### Step 3: 리라이트

**보존 (건드리지 않는 것):**
- HTML 구조, 클래스, 속성
- `<table>`, `<code>`, `<pre>` 블록
- 이미지, figcaption
- stat-card 숫자
- PebblousPage.init config
- FAQ (config.faqs)
- `<title>`, `<meta>` 태그
- 숫자·팩트 자체 (톤만 변환, 내용 변경 금지)

**변환:**
- 긴 나열형 → 짧은 선언 + 긴 사색 교차
- 논문체 어미 → 에세이체 (-다, -했다, -이다)
- 무인칭 → 1인칭 관찰자 ("열어보면", "따라가 보면")
- 냉소적 서술 → 따뜻한 관찰 ("없다"의 반복 대신 맥락 부여)
- 추상 서술 → 구체 숫자/이름
- 적절한 곳에 질문 삽입

### Step 4: 자기 검증
```bash
grep -c "본 보고서\|규명한다\|분석하고자\|살펴본다" [file]  # 0이어야
grep -c "혁신적\|획기적\|놀라운" [file]  # 0이어야
```
+ "짧은 문장만 연속 3개 이상"인 구간이 있으면 긴 사색 문장 추가

### Step 5: before/after 3건 보고
리라이트 전후 대표 문단 3개를 사용자에게 보여주고 톤 확인.
